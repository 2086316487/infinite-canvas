# 方案三：图片异步任务 + 前端轮询

## 背景

实际排查中出现过这种现象：中转站显示生成成功并扣费，Render 也拿到上游结果，但工作台显示失败。常见原因不是中转站失败，而是同步链路太长：

```text
Browser -> Render Go API -> AI upstream -> Render writes large response -> Browser
```

当生成耗时较长、浏览器连接波动、响应体过大或写回阶段出现 `broken pipe` 时，上游任务已经成功，但前端没有拿到最终响应。

方案三把链路拆成两步：

```text
1. Browser -> Render: create image task, quickly returns task id
2. Browser -> Render: poll image task until succeeded or failed
```

图片生成在 Render 后端任务中继续执行，不再依赖浏览器维持一条长连接。

## 当前已落地范围

- 新增 `image_tasks` 数据表，由 GORM AutoMigrate 创建。
- 新增图片任务接口：
  - `POST /api/v1/image-tasks/generations`
  - `POST /api/v1/image-tasks/edits`
  - `GET /api/v1/image-tasks/{id}`
- 后端创建任务后快速返回 task id，并由 goroutine 执行上游请求。
- 上游成功后会先把原始响应写入 recovery 清单；data URL 结果会转存为服务端本地图片文件，http(s) 结果 URL 会直接写回任务记录，避免 Render 在上游已成功扣费后又因二次下载结果图而卡住或重试。
- 上游失败后记录错误并尝试退还积分。
- 后端增加基于数据库字段的租约队列：`locked_by`、`locked_until`、`attempts`、`next_run_at`。
- Render 服务启动后会立即扫描可执行任务，并持续轮询领取到期的 `pending` / `running` 任务。
- Worker 会定时续租运行中的任务；短暂网络错误、429 或 5xx 会按次数有限重试。
- 新增 `APP_ROLE=all|web|worker` 启动角色，可把图片任务 worker 从 Web 进程中拆开独立运行。
- 图片工作台远程模式已改为创建任务并轮询。
- 画布远程生图已保存 `imageTaskId` / `imageTaskMode` 到图片节点，刷新后可继续轮询。
- 画布助手远程生图消息已保存 `imageTasks` / `imagePrompt`，刷新后可继续轮询并回填助手图片。
- 画布批量图片生成会汇总 child 成功、失败、等待状态，并同步更新 batch root 节点主图与状态摘要。

## 后端接口

### 文生图任务

```http
POST /api/v1/image-tasks/generations
Authorization: Bearer <user-token>
Content-Type: application/json
```

请求体沿用 OpenAI 兼容图片接口格式：

```json
{
  "model": "gpt-image-2",
  "prompt": "...",
  "n": 1,
  "size": "1024x1024",
  "quality": "auto",
  "response_format": "url",
  "output_format": "png"
}
```

### 图生图 / 修图任务

```http
POST /api/v1/image-tasks/edits
Authorization: Bearer <user-token>
Content-Type: multipart/form-data
```

表单字段沿用 `/images/edits`：

- `model`
- `prompt`
- `n`
- `size`
- `quality`
- `response_format`
- `output_format`
- `image`
- `mask`

### 查询任务

```http
GET /api/v1/image-tasks/{id}
Authorization: Bearer <user-token>
```

状态集合：

```text
pending
running
succeeded
failed
cancelled
```

## 画布批量 root 汇总规则

批量生图 root 节点不再只看最后一次请求是否抛错，而是扫描所有 child 节点：

- 全部成功：root 标记为 `success`，清空错误摘要，并选择第一张成功图作为主图。
- 部分成功：root 标记为 `success`，保留可用主图，并显示成功/失败数量摘要。
- 仍有任务运行：root 保持 `loading`，如果已有失败则显示当前失败和等待数量。
- 全部失败：root 标记为 `error`，显示全部失败摘要。

这能避免批量远程任务中“一个 child 失败拖垮整批”或“失败被误判成功”的问题。

## 当前隐患

- 已支持把 worker 通过 `APP_ROLE=worker` 独立部署；但默认 `APP_ROLE=all` 仍是 Web/API 与 worker 同进程运行，若线上未拆分部署，服务重启或滚动发布时仍会共享同一实例生命周期。
- 如果实例在上游已经成功但 recovery 清单尚未来得及写成时崩溃，极端情况下仍可能触发重复请求；清单写成后会优先恢复本地结果，不再重新请求上游，恢复链中的临时落盘失败也会进入同一套有限重试或失败终态。
- data URL 结果默认写入服务端本地 `data/generated-image-tasks/`，并以确定性文件名保存；http(s) 结果 URL 暂不经 Render 二次下载，直接保存到任务 outputs。没有对象存储前，本地文件仍可能在重启、扩容或切实例后丢失；远程 URL 也可能受上游有效期限制。
- recovery 清单默认写入 `data/generated-image-task-recovery/`，不会通过 `/api/media/generated-images/:id` 对外提供；公开媒体接口只服务图片扩展名。
- 结果尚未转存到对象存储，当前 data URL 结果仍依赖单机持久化磁盘，http(s) 结果依赖上游 URL 可访问性。

## 后续升级建议

优先升级两点：

1. 后端结果转存对象存储
   - 当前 data URL 结果会落到本地稳定文件并补 recovery 清单，http(s) 结果会直接写回 outputs；下一步可统一上传到 Supabase Storage 或其他对象存储。
   - `outputs` 可保存稳定 storage URL 或 asset id。
   - 进一步解决多实例不共享本地磁盘、单机磁盘丢失和跨实例访问问题。

2. 独立 worker 进程
   - 已支持通过 `APP_ROLE=worker` 启动独立 worker 进程。
   - Web API 可改用 `APP_ROLE=web`，只负责创建任务和查询结果。
   - 单独的 worker 进程负责扫描、领取、执行、重试和写结果。
   - 进一步降低 Render Web 服务重启、扩容或滚动发布时的重复执行风险。

## 推荐部署角色

单实例兼容模式：

```text
APP_ROLE=all
```

适合本地或简单部署，Web/API 和图片任务 worker 跑在同一个 Go 进程里。

分离部署模式：

```text
Web Service: APP_ROLE=web
Worker:      APP_ROLE=worker
```

推荐线上拆分成两个进程：

- Web Service 负责页面、接口、鉴权和任务创建。
- Worker 负责轮询任务表、领取任务、调用上游、写回结果和自动重试。

如果使用 Docker Compose，项目根目录已经提供可直接叠加的 override：

```bash
docker compose -f docker-compose.yml -f docker-compose.worker.yml up -d
```

本地源码构建版本：

```bash
docker compose -f docker-compose.local.yml -f docker-compose.local.worker.yml up -d --build
```

## 验收标准

- 创建任务接口能快速返回 task id。
- 图片工作台、画布节点和画布助手消息能立即进入生成中状态。
- 页面刷新后，未完成的图片任务可以继续轮询，画布助手消息可在完成后回填生成图片。
- 上游成功但浏览器曾断开时，重新打开仍能查到 `succeeded` 和 outputs。
- 任务完成后 `outputs` 返回站内图片 URL（如 `/api/media/generated-images/...`）或上游 http(s) URL；刷新页面后仍应能展示，画布无法把远程 URL 保存到本地时也不应把已成功生成的任务标成失败。
- 模拟 recovery 清单已存在但数据库尚未写回时，worker 重启后应恢复结果；如果 data URL 本地落盘短暂失败，应按任务重试节奏重试，而不是长期停在 `running`。
- 短暂网络错误、429 或 5xx 不会立刻终态失败，而会在有限次数内自动重试。
- 批量画布 root 能准确显示全部成功、部分成功、全部失败。
- Render 日志可按 task id 串联排查 `start / succeeded / failed`。
